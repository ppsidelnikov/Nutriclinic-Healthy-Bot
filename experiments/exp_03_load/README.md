# Нагрузочное тестирование (§3.3)

## Цель

1. Определить **точку перегиба** системы (R6 RAG + V6 photo) — максимальное число
   одновременных пользователей, при котором p95 latency ≤ 5 сек, error rate ≤ 1 %.
2. Замерить **фактическое потребление CPU / RAM** контейнеров под нагрузкой.
3. Обосновать **рекомендации по серверу** для продакшен-развёртывания.

## Методика

### Архитектура нагрузки

Бот общается через Telegram polling, поэтому HTTP-нагрузочник не подходит.
Вместо этого тестируем **сервисный слой бота напрямую**: запускаем N параллельных
«виртуальных пользователей», каждый раз в ~10 секунд выполняет один из сценариев:

- **70 %** — текст (RAG-путь: профиль + история + R6-поиск)
- **25 %** — фото блюда (V6 двухпроходный pipeline)
- **5 %**  — лёгкие операции (профиль, история)

### Ramp-test

Скрипт `ramp_test.py` последовательно прогоняет нагрузку с растущим числом
пользователей: **1 → 5 → 10 → 25 → 50 → 100**. На каждом уровне:

- 3 минуты стабильной фазы
- параллельно `monitor.py` пишет `docker stats` каждую секунду в CSV
- после уровня — 30-секундный cooldown для разгрузки Redis-кэша

### Метрики

| Метрика | Что значит |
|---|---|
| **RPS** | Реальная пропускная способность (выполненных операций / сек) |
| **p50/p95/p99 latency** | Перцентили задержки одного хода (отдельно text и photo) |
| **error rate** | % операций с исключением или таймаутом |
| **CPU peak %** | Пиковая нагрузка процессора каждого контейнера |
| **MEM peak MB** | Пиковое потребление оперативной памяти каждого контейнера |

### Точка перегиба

Уровень нагрузки, при котором **впервые нарушается SLA**:
- p95(text) > 5000 мс ИЛИ
- p95(photo) > 5000 мс ИЛИ
- error rate > 1 %

### Сайзинг сервера

Берём максимальный стабильный уровень (последний, прошедший SLA), смотрим
пиковое потребление, добавляем 50 % запас и округляем вверх:

```
Рекомендуемые vCPU = ceil( bot_CPU_peak% / 100 × 1.5 ) + 1
Рекомендуемая RAM  = (bot_MEM_peak + pg_MEM_peak) × 1.5, округлено до 0.5 GB
```

## Запуск

### 1. Поднять стек

```bash
cd ~/Dev/Nutriclinic-Healthy-Bot
docker compose up -d
docker compose logs -f bot   # дождись "RAG прогрет"
```

### 2. Установить зависимости для нагрузочника локально

Нагрузочник работает локально (вне Docker), импортирует код бота напрямую,
ходит в Postgres/Redis через `localhost:5432` / `localhost:6379`.

```bash
pip3 install -r requirements.txt   # из корня бота
```

### 3. Полный ramp-test (рекомендуется)

```bash
cd experiments/exp_03_load
python3 ramp_test.py
```

Прогон ~25 минут (6 уровней × 3.5 мин с cooldown). На выходе:
- `results/results_ramp_n*.jsonl` — операции каждого уровня
- `results/stats_ramp_n*.csv` — docker stats каждого уровня
- `results/ramp_summary.json` — индекс файлов

### 4. Анализ и рекомендации

```bash
python3 recommend.py
```

Выводит сводную таблицу всех уровней + точку перегиба + рекомендации по vCPU/RAM.
Сохраняет `results/recommendation.txt` для §3.3 диссертации.

## Дополнительные сценарии (§2.3 — L1–L4)

После ramp-теста можно прогнать четыре сценария архитектурных оптимизаций:

```bash
# L1 — базовая конфигурация (всё включено)
python3 load_test.py --scenario L1 --users 50 --duration 600

# L2 — без Redis-кэша FatSecret (отключаем кэш через env)
FATSECRET_CACHE_DISABLED=1 python3 load_test.py --scenario L2 --users 50 --duration 600

# L3 — последовательная (не параллельная) загрузка контекста
SERIAL_CONTEXT_OPS=1 python3 load_test.py --scenario L3 --users 50 --duration 600

# L4 — pgvector с IVFFLAT вместо HNSW (требует миграцию индекса)
python3 load_test.py --scenario L4 --users 50 --duration 600

python3 compare_scenarios.py
```

## Файлы

- `load_test.py` — нагрузочник на asyncio
- `monitor.py` — запись docker stats в CSV
- `ramp_test.py` — оркестратор: load + monitor на нескольких уровнях
- `recommend.py` — анализ и рекомендации по железу
- `compare_scenarios.py` — сравнение L1–L4 (для §3.3)
